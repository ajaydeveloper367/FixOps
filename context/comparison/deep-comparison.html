Gaps that remain — specific to your requirements

Things neither project has today, even after reading the full source code.

Airflow integration — both missing

HolmesGPT has no Airflow toolset (Prefect MCP exists). IncidentFox has it only as a UI demo mock — no backend skill, no DAG API integration. You need to build this as a new MCP adapter (HolmesGPT path) or a new skill (IncidentFox path).

MongoDB investigation — IncidentFox gap

HolmesGPT has two native Mongo paths (wire protocol + Atlas). IncidentFox has neither — MongoDB is only mentioned in their RAG/docs layer. If you pick IncidentFox, Mongo inspection needs a new skill built from scratch.

Two-layer metadata model — both missing

Neither has your Inventory (routing metadata) + Dataflow Graph (ownership resolution) model. HolmesGPT has no topology layer at all. IncidentFox has auto-topology discovery but it's not a queryable structured schema that the controller can use for deterministic routing.

Rules-first routing — both rely on LLM

HolmesGPT: single LLM picks all tools. IncidentFox: config-driven at top level but LLM decides subagent + tool selection inside session. Your "LLM extracts entities only, rules decide worker" principle needs a custom routing layer in both cases.

Confidence scoring + structured output contract

Neither worker returns a JSON schema with findings, evidence_refs, confidence, ruled_out, next_suggested_check. Both return narrative prose. Your controller's stop-on-high-confidence logic requires this to be built as a wrapper around workers in either project.

Staged context loading (3-stage budget)

HolmesGPT has per-tool memory limits. IncidentFox has stats-first sampling per skill. Neither enforces your Stage 1 → Stage 2 → Stage 3 budget with hard token limits across the investigation as a whole.

Post-action verification loop

Neither has a built-in "execute → wait → verify recovery" loop. HolmesGPT has no approval gate. IncidentFox approves and executes but stops there — no automated confirmation that failures dropped or backlog cleared.

Scale strategy: DAG graph auto-extraction

Neither has tooling to auto-derive a dependency graph from DAG Python code + app configs + Helm values at your scale. HolmesGPT's GitHub MCP can read repos but won't build a structured graph. This is your original piece regardless of which base you use.

Revised recommendation (after reading source code)

IncidentFox base

Still the closer architectural fit. Multi-agent config, Slack-first, strong GitHub/Confluence integration, real K8s gate via k8s_agent, Loki + Prometheus + Kafka all present. Main gaps to fill: Airflow skill (new), Mongo skill (new), Tempo→Jaeger migration or Tempo skill, rules-first routing wrapper, confidence score output contract.

HolmesGPT for tools

Better as a toolset/adapter reference. Its native Mongo wire protocol, Loki, Tempo (Grafana), Kafka, Confluence, and Prometheus toolsets are clean and production-grade. Consider porting these tool implementations as MCP adapters if building on IncidentFox's framework. The single-agent architecture is not worth adopting.

Key clarification

The earlier statement "neither has Kafka" was wrong — both have Kafka admin/inspection. The real gaps are: Airflow (both), MongoDB investigation in IncidentFox, and all the architectural layers (routing engine, metadata model, confidence scoring, staged context). Those are yours to build regardless.
