What neither project covers — your unique gaps

Airflow + Kafka + Data Pipeline
Neither has an Airflow worker, Kafka consumer lag awareness, or understanding of the Mongo → Airflow → Kafka → K8s → pdata chain. Your entire data pipeline layer is a gap in both projects.
Two-layer metadata model
Your Inventory + Dataflow Graph separation (deterministic routing vs semantic ownership) does not exist in either project. Both infer topology dynamically — which your design explicitly rejects for routing.
Rules-first routing engine
Both use LLM to decide tool/worker selection. Your principle that "LLM extracts entities, rules decide worker" requires a custom deterministic routing layer that neither provides.
Scale strategy: DAGs
Neither has a strategy for auto-deriving a dependency graph from DAG code + app code + Helm configs. Your hybrid model (thin manual inventory + auto-derived graph + RAG) is an original contribution.
