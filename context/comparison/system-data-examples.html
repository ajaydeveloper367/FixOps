Dataflow Graph — Visual & Schema

The graph resolves ambiguous alerts: "who writes to pdata?" → follow edges to find the responsible service.

BulkFHIR Processing Pipeline — Full Flow

Nodes schema (how nodes are stored)
-- Postgres table: graph_nodes id: pdata node_type: collection display_name: pdata metadata: { database: processing_db, mongo_cluster: mongo-cluster-A, primary_owner: bulkFHIR-processor } inventory_ref: pdata # → join to inventory -- Another node id: bulkFHIR-events node_type: topic metadata: { kafka_cluster: kafka-prod-1, partitions: 12, consumer_group: bulkFHIR-processor-group }
Edges schema (how edges are stored)
-- Postgres table: graph_edges from_id: bulkFHIR-ingest-dag to_id: bulkFHIR-events edge_type: produces label: publishes processed records from_id: bulkFHIR-processor to_id: pdata edge_type: writes_to label: status updates (pending→completed/failed) from_id: bulkFHIR-events to_id: bulkFHIR-processor edge_type: consumed_by label: consumer group: bulkFHIR-processor-group -- Edge types: produces, consumes, writes_to, -- reads_from, triggers, depends_on, owned_by
How controller queries the graph — "who writes to pdata?"
-- Query 1: find direct writer SELECT from_id, edge_type FROM graph_edges WHERE to_id = 'pdata' AND edge_type IN ('writes_to', 'produces') → Result: bulkFHIR-processor (writes_to) -- Query 2: get 1-hop upstream of bulkFHIR-processor SELECT from_id, edge_type FROM graph_edges WHERE to_id = 'bulkFHIR-processor' AND edge_type = 'consumed_by' → Result: bulkFHIR-events (Kafka topic feeds the processor) -- Query 3: what produces bulkFHIR-events? SELECT from_id FROM graph_edges WHERE to_id = 'bulkFHIR-events' AND edge_type = 'produces' → Result: bulkFHIR-ingest-dag (Airflow DAG) -- Full upstream chain now known without loading any logs/metrics: pdata ← bulkFHIR-processor ← bulkFHIR-events ← bulkFHIR-ingest-dag ← mongo-source

Controller Decision Logic

Step 1 — Routing rules (deterministic, no LLM)

Extracted entity typeRouting pathFirst worker calledExample
pod / node / deployment Infra direct — no lookup needed K8s / Infra Worker Alert: "dp-engine pod CrashLoopBackOff prod-eks-1" — cluster known from alert, goes straight to K8s worker
service / job name Inventory lookup → get runtime_type + cluster Worker matching runtime_type Alert: "bulkFHIR-processor failures" → inventory → runtime: kubernetes, cluster: prod-eks-2 → K8s Worker
DAG name Inventory lookup → get airflow_instance Pipeline / Airflow Worker Query: "bulkFHIR_ingest_dag failed" → inventory → airflow-prod URL + creds → Airflow Worker
collection / table / topic Dataflow graph lookup → resolve owner service → Inventory Worker for owner service Alert: "pdata failed messages" → graph → owner: bulkFHIR-processor → inventory → K8s Worker
unclear / ambiguous Fallback: observability labels → log tags → past incidents Observability Worker first Alert: "processing error count high" (no entity) → check Loki labels → infer service → then route normally

Step 2 — Context budget (what gets loaded at each stage)

StageWhat is loadedCostCondition to proceed to next
Stage 1 Alert payload · Matched inventory record · 1-hop graph neighborhood · Recent deploy/config change · Basic pod/DAG health status ~500–1k tokens Confidence < 0.8 after Stage 1 → proceed to Stage 2
Stage 2 Targeted logs (service + time range, max 200 lines) · Related KPIs last 30 min · DAG run history · Kafka consumer lag · DB error rates ~2k–5k tokens Confidence < 0.75 after Stage 2 → proceed to Stage 3
Stage 3 Code snippets (relevant module only) · Configs · Runbook / Confluence page · Distributed traces · Past incident history ~5k–15k tokens Only when Stage 2 is inconclusive. Never loaded by default.

RCA generated — confidence 0.96 ≥ 0.85 → stop escalating T+119s
ROOT CAUSE: dp-service fetches BATCH_SIZE=5000 messages per Kafka poll and materializes all 5000 enriched objects in heap simultaneously (MessageProcessor.java:87) before writing any to MongoDB. Each enriched object is ~160KB → ~800MB allocation per batch cycle. With raw messages still in scope + JVM overhead, peak heap reaches ~1.3GB, exceeding the 2Gi container memory limit → OOMKill. EVIDENCE CHAIN: ConfigMap BATCH_SIZE=5000 → Code: collect(toList()) materialises full enriched batch before bulkWrite → Loki: "Enriching 5000 messages" log → heap spikes from 312MB to 1876MB → Prometheus: 300MB→1.95GB spike every 30s (Kafka poll interval) → K8s: OOMKill at enrichment step, not during poll CONFIDENCE: 0.96
8
Two remediation options — approval required for any action
Option A — Quick fix (30 min)
Increase memory limit from 2Gi to 4Gi in the deployment manifest.

Pros: Stops OOMKill immediately. No code change, no re-test needed. Safe rollout with rolling update.

Cons: Does not fix root cause. If message size grows or BATCH_SIZE is increased, OOM will return. Costs ~$X/month in extra node memory.

Action: Edit dp-service/k8s/deployment.yaml → resources.limits.memory: 4Gi → kubectl apply
Option B — Root cause fix (2–3 days)
Set DP_BATCH_SIZE=500 in ConfigMap AND refactor MessageProcessor.java to process in chunks of 500 within the poll loop — stream write each chunk to Mongo before loading the next.

Pros: Addresses root cause. Peak heap drops to ~130MB per chunk. Scales safely with message volume growth.

Cons: Requires code review + test + deploy. Slightly higher Kafka offset commit frequency.

Expected result: Memory stabilises at 400–500MB, no OOMKills.

Why K8s scheduler made this decision — root cause explained T+96s
ROOT CAUSE: Three memory-intensive services (bulkFHIR-processor, dp-service, analytics-worker) are co-scheduled on the same nodes because their Kubernetes resource requests are extremely under-provisioned (total 840Mi = 11.2% of node). The K8s scheduler packs pods using requests as the placement signal, not actual usage. When all three hit peak usage simultaneously during the data processing window (14:45–15:30 UTC daily), combined actual memory (6.58Gi) exceeds node capacity → eviction cascade. SCHEDULER DECISION LOGIC (why it packed them): K8s sees available node memory = 7.5Gi − 840Mi (requests) = 6.66Gi "free" → places all 3 services: "plenty of headroom" → BUT actual peak usage is 6.38Gi — almost no real headroom at all OVER-COMMIT RATIO: limit / request = 5.5Gi / 840Mi = 8.6× (safe range for production: 1.5–3×) actual / request = 6.58Gi / 840Mi = 7.8× (actual usage vs what scheduler reserved) NOTE: This is expected K8s behaviour — the scheduler is working correctly. The problem is the requests are wrong, not the scheduler. Auto-remediation not applicable here — this requires deliberate admin decisions.
