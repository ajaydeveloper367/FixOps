Infrastructure Probe Agent

Core Tool Adapters

Hit partner status page or well-known status URL. Parse for outage indicators. Control check: also probe a known-healthy endpoint (api.github.com) to rule out our egress.

What Goes into the RAG Store

Confluence runbooks
Troubleshooting guides per service: "How to debug bulkFHIR-processor OOM", "Airflow DAG failure checklist". Chunked by section heading.
Past RCA reports
Previous incident resolutions stored as structured text. Semantic match: "same OOM in dp-service 3 months ago → fix was BATCH_SIZE reduction". Highest value.
Architecture docs
Service descriptions, data flow narratives. Helps worker understand upstream/downstream relationships when graph is ambiguous.
Code comments / README
Service-level README explaining config options: "BATCH_SIZE: number of messages per Kafka poll. Memory impact: ~160KB per message."

When RAG is Triggered

Stage 1 always
Query: "{service_name} {alert_class}". Injects general troubleshooting context upfront, before first worker call.
After each worker result
Query refined with worker findings: "oom batch size kafka poll memory". Retrieves more targeted runbook sections for next stage.
RCA synthesis
Query past RCA reports semantically: finds similar incidents. Injects "similar past resolution" as additional evidence in the final RCA.
NOT triggered when
Budget < 500 tokens remaining, or similarity score < 0.75 (no relevant chunks). Fails gracefully without impacting investigation.

Other Token Control Mechanisms

📉 Observability data summarisation
Prometheus time-series is not dumped raw. Adapter computes: min, max, p95, spike timestamps. 500 raw data points → 4 numbers. Saves ~2,000 tokens per metric.
✂ Log pre-filtering
Loki adapter filters to ERROR/WARN lines only, max 50 lines, most recent first. Strips ANSI codes, stack trace deduplication. 5,000 log lines → 50 relevant lines.
🗜 Graph serialisation
Dataflow graph sent as compact adjacency JSON: {"bulkFHIR-processor": {"writes": ["hl7message"], "reads": ["bulkFHIR-events"]}}. Not Cypher or full node objects.
💾 Result caching
K8s pod health result cached in Redis for 5 min per pod. If two workers need the same data, second call hits cache. No duplicate API calls or LLM reprocessing.

Confidence Score — What It Is, How It's Calculated, How Decisions Are Made

The confidence score is the single number that drives every escalation decision in Infinity-Sentinel. It is not a guess — it is a reasoned, auditable output produced by each worker and verified by programmatic rules.

Score Levels at a Glance

0.0 – 0.45
Very Low
Single source, vague
0.45 – 0.65
Low
Symptom known, cause unclear
0.65 – 0.84
Medium
Strong signal, needs confirmation
0.85 – 1.0
Conclusive ✓
Multi-source, all alternatives ruled out

What Moves the Score

+0.20
Error confirmed in logs AND matching metric spike
OOMKill in Loki + Prometheus memory spike at same timestamp
+0.20
Config/code directly confirms the hypothesis
ConfigMap: DP_BATCH_SIZE=5000, Code: collect(toList()) materialises all 5000
+0.25
External authoritative confirmation
Partner status page: 🔴 Major Outage confirmed — third-party API down
+0.15
Multiple alternative causes ruled out explicitly
Bad deploy ✗, network issue ✗, broker failure ✗, pod crash ✗
−0.20
Only one data source available
Only Kubernetes data, no logs or metrics to corroborate
−0.20
Finding is vague or general
"Memory issue" vs "BATCH_SIZE=5000 causes ~800MB heap allocation"
−0.15
Unexplored alternative explanations exist
"Could also be a GC configuration issue or memory leak elsewhere"
−0.20
Contradicting signals across sources
Logs say OOM but memory metric shows only 50% usage

Programmatic Overrides — Why the Score Is Auditable, Not a Black Box

The LLM worker produces an initial confidence score. The controller then applies deterministic rules on top — making the final score explainable and testable.

# controller/confidence.py — applied after every worker LLM call def apply_overrides(score: float, findings: list, evidence: dict, stage: int) -> float: # Hard floor: can't be high-confidence with nothing ruled out if len(evidence.get("ruled_out", [])) == 0: score = min(score, 0.65) # Hard floor: single data source at Stage 1 if stage == 1 and len(evidence.get("tools_called", [])) <= 1: score = min(score, 0.70) # Boost: external authoritative source confirms outage if evidence.get("external_status_page_down") and "ConnectTimeout" in str(findings): score = max(score, 0.95) # partner outage confirmed → very high confidence # Boost: specific config value found that matches symptom if evidence.get("config_value_confirmed") and evidence.get("code_pattern_confirmed"): score = max(score, 0.92) # config + code = strong root cause # Penalty: contradicting signals if evidence.get("contradicting_signals"): score = min(score, 0.60) return round(score, 2)

HolmesGPT vs IncidentFox vs Infinity-Sentinel — Feature Comparison

Based on source code analysis of both open-source projects. "None" means the feature is genuinely absent — not just undocumented. Our proposed design deliberately takes the best of both and adds the missing pieces.

HolmesGPT
Single-agent LLM troubleshooter. Issues tool calls in a loop until it reaches a conclusion. No orchestration layer — one agent does everything. Strong on Kubernetes + Prometheus toolsets. MCP-native.
⚠ Good tool coverage, but single agent — no specialization, no staged context, no confidence gating, no inventory graph.
IncidentFox
Multi-agent SRE framework. Orchestrator routes to specialised skill agents. Has credential proxy, MCP tools, Slack integration. Built for structured incident workflows.
⚠ Best multi-agent architecture of the two, but no confidence scoring, no RAG, no token budget, no inventory/graph layer.
Infinity-Sentinel (Proposed)
Takes IncidentFox's multi-agent orchestration + HolmesGPT's tool adapters and adds: confidence gating, staged context loading, RAG, inventory + dataflow graph, token budget, PagerDuty-first.
🎯 Combines the best of both and adds the architectural pieces missing for production-grade reliability at scale.

Why We Build Our Own — Key Reasoning

From IncidentFox Multi-agent structure
IncidentFox's orchestrator + specialised skill-agent model is the right architecture. Workers that know only one domain (K8s, Airflow, Kafka) produce better, tighter prompts and smaller contexts than one agent trying to know everything. We take this pattern entirely.
From HolmesGPT Tool adapter depth
HolmesGPT has the best breadth of tool adapters (K8s, Prometheus, Loki, Kafka, GitHub, etc.). The MCP-native approach and datasource-catalog concept are sound. We adopt the adapter pattern but wrap them in typed Python classes instead of raw MCP calls.
New: Confidence + Staged Context
Neither OSS project has confidence scoring or staged context loading. Without it, investigations either over-spend tokens (load everything upfront) or under-investigate (stop too early). The 0.85 threshold + 3-stage escalation is the key architectural addition that makes the system cost-efficient and thorough.
New: Inventory + Dataflow Graph
Both OSS projects rely on the LLM to figure out what to query and where. This wastes tokens and produces inconsistent results. A queryable inventory DB + dependency graph means the controller knows in 1 DB lookup that "hl7message is written by bulkFHIR-processor consuming bulkFHIR-events" — before any LLM call.
New: RAG over runbooks
Neither project has semantic retrieval over institutional knowledge. When an OOM happens, a 300-token Confluence chunk saying "dp-service batch size controls memory" is more valuable than 3,000 tokens of raw logs. RAG bridges the gap between what the LLM knows generically and what the engineering team has documented specifically.
Not replacing either entirely
HolmesGPT's tool adapters can be re-used as-is (wrapped as our adapters) rather than rewriting from scratch. IncidentFox's credential proxy architecture is worth adopting directly. Building Infinity-Sentinel is additive — we compose on top of proven pieces rather than discarding them.
